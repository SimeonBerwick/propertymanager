# Property Manager V1 - Onboarding & Invite System Spec

## Purpose
This document defines the onboarding and invite model for Property Manager V1 based on the chosen rollout direction:
- the operator is the paying customer
- tenant access is free
- vendor access is free
- tenant/vendor users join the operator's workspace through operator-controlled invites or access codes

This is not just account creation UX. It is part of the trust boundary.

---

## 1. Product direction

### Commercial model
- Operator / landlord / property manager pays for the product
- Tenant access is free as part of the operator's workspace
- Vendor access is free as part of the operator's workspace
- Tenant and vendor roles are participant surfaces, not standalone paid products

### Strategic reason
This matches the real buyer.
The operator gets the operational value and owns the workspace.
Tenants and vendors participate because the operator needs them in the workflow, not because they are separate customers to monetize directly.

---

## 2. Design principles

### Principle 1 - Operator owns the workspace
The operator creates and controls the organization.
Tenant/vendor access exists only inside that operator-owned boundary.

### Principle 2 - Onboarding must reduce friction
Tenants and vendors should not need to "figure out the product" before they can use it.
Joining should feel like following a clear invitation, not discovering a SaaS on their own.

### Principle 3 - Invite system is security-sensitive
A code/link is not just a convenience feature.
It controls who can join which organization, unit, or vendor relationship.
Treat it as a permissions and data-exposure surface.

### Principle 4 - Default to operator-generated invites
Do not rely on unaffiliated self-signup for tenants/vendors.
Operator-issued access should be the default path.

---

## 3. Recommended onboarding model

### Operator onboarding
Operators create the paid account and initial organization.

#### Flow
1. Operator signs up for paid trial/plan
2. Operator creates organization/workspace
3. Operator creates first property / unit structure
4. Operator begins inviting tenants and vendors

#### Required outcomes
- operator account exists
- organization exists
- operator lands in a usable dashboard
- onboarding quickly points toward first real setup tasks

---

### Tenant onboarding
Recommended default: **property/unit-scoped invite link or access code** generated by the operator.

#### Why this default
- keeps tenant access tied to the right property/unit context
- reduces setup mistakes
- makes it easier to scope visibility correctly
- feels natural for residents: "use this link/code for your building/unit"

#### Flow
1. Operator generates tenant invite for a property/unit
2. Tenant opens invite link or enters code
3. Tenant confirms identity details
4. Tenant creates password / signs in
5. Tenant lands on resident request/status surface already connected to the right unit context

#### Tenant invite requirements
- scoped to one organization
- scoped to one property/unit or tenant record
- time-limited or revocable
- single-use or safely re-usable based on design choice
- clear failure states for expired/invalid/already-used invites

---

### Vendor onboarding
Recommended default: **vendor company invite link or access code** generated by the operator.

#### Why this default
- vendors are usually attached at the organization level, not a single unit
- keeps vendor access tied to the operator's approved vendor relationship
- makes it easier to scope assigned work and avoid free-floating vendor signup

#### Flow
1. Operator creates vendor record or selects existing vendor
2. Operator generates vendor invite
3. Vendor opens invite link or enters code
4. Vendor creates password / signs in
5. Vendor lands in vendor queue limited to assigned, vendor-visible jobs

#### Vendor invite requirements
- scoped to one organization and one vendor record
- revocable
- time-limited
- clear failure states
- does not grant access to unrelated vendor records or requests

---

## 4. Recommended default join model

### Default recommendation
Use a **hybrid model**:
- **tenant:** property/unit-scoped invite link or access code
- **vendor:** vendor-company-scoped invite link or access code

### Why hybrid is best
Tenant and vendor relationships are structurally different.
Trying to force both through one identical invite model will create awkward compromises.

---

## 5. Access code vs invite link

### Invite links
**Pros**
- lowest friction
- easiest on mobile
- best for real-world rollout

**Cons**
- forwarding risk if not designed carefully
- must handle expiry, revocation, and one-time use properly

### Access codes
**Pros**
- easy to communicate over text/email/phone
- useful when links are inconvenient
- can be paired with identity confirmation

**Cons**
- more manual friction
- easier for users to mistype

### Recommendation
Support both eventually, but for V1 rollout:
- **primary:** invite links
- **fallback:** access codes

That gives the smoothest real-world onboarding without losing operational flexibility.

---

## 6. Data model implications

The invite system likely needs dedicated models such as:
- `Invite`
- `InviteType` (TENANT, VENDOR, possibly OPERATOR_TEAM later)
- `organizationId`
- `propertyId` nullable
- `unitId` nullable
- `vendorId` nullable
- `tenantId` nullable
- `tokenHash` or secure code hash
- `expiresAt`
- `revokedAt`
- `usedAt`
- `createdByUserId`

### Important rule
Never store raw invite secrets in a casually readable way if they are long-lived or high-trust.
Hash where appropriate.

---

## 7. Security requirements

- invite must be scoped to the correct org/entity
- invite must expire
- invite must be revocable
- invite consumption must be auditable
- used invites should not stay broadly reusable unless explicitly designed that way
- invalid/expired/revoked invite states must fail cleanly
- invite acceptance must not allow cross-org or cross-record escalation

---

## 8. UX requirements

### Operator UX
- generate invite from the relevant workflow, not a buried admin page
- copy link/code easily
- see invite status: active, used, expired, revoked
- revoke and re-issue cleanly

### Tenant UX
- very short path from invite to usable request screen
- plain language, not admin language
- mobile-first
- clear messaging if invite is invalid or expired

### Vendor UX
- short path from invite to assigned jobs
- clear understanding of what vendor can and cannot see
- no unnecessary product complexity before first useful action

---

## 9. Rollout sequence

### Phase 1 - pilot-ready
- operator paid onboarding exists
- tenant invite flow exists
- vendor invite flow exists
- invites are scoped, revocable, and expiring
- joined users land in the correct surfaces

### Phase 2 - paid beta-ready
- polished invite management UX
- resend / revoke / regenerate flows
- support recovery for failed/expired invites
- audit trail for invite creation and acceptance

### Phase 3 - publicly sellable
- fully polished onboarding funnel
- self-serve operator signup tied into billing
- support docs and recovery flows
- analytics on invite completion and activation rates

---

## 10. Recommendation summary

### Chosen direction
- operator pays
- tenant is free
- vendor is free
- invite-controlled onboarding is the default

### Best default implementation
- operator creates the workspace
- tenant joins with property/unit-scoped invite link
- vendor joins with vendor-company-scoped invite link
- access codes exist as fallback, not the primary path

### Why this is the right move
It matches the real buyer, reduces friction, keeps security boundaries clearer, and avoids pretending tenant/vendor roles are separate paid products.
